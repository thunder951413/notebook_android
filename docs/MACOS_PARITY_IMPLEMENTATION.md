# macOS 最近改动的 Android 对齐说明

本文记录 Android 端对照 Notebook macOS 端实现的功能、协议与边界，供后续维护和回归测试使用。

## 已对齐功能

### 列表与长笔记加载

- 列表查询只读取标题、摘要、时间、目录、提醒、版本和同步状态，不读取正文、同步快照或冲突快照。
- 选择笔记后才加载正文；SQLite 大字段按 64 KiB 分段读取，避免 CursorWindow 超限。
- 阅读模式在后台把 Markdown 解析为标题、段落、列表、引用、表格、代码围栏和空白等语义块，`LazyColumn` 只创建可见块。代码围栏、表格和列表不会再被固定字符窗口从中间切断。
- 超长普通段落优先在换行边界继续拆分；稳定块 ID 由块类型、标题路径和内容前缀生成，在前文插入内容后仍可用于列表复用和目录定位。
- 正式编辑时仍加载完整正文，以保证查找、选择和修改语义正确；窗口化只影响阅读渲染，不改变存储内容。
- 超过 512 行的正文同步到 macOS 块文档时，每 256 行合并为一个段落块，正文可无损往返，也避免产生数万小块。块 ID 在相同笔记和位置上稳定。

### 阅读位置持久化与跨端同步

- Room 表：`reading_positions`。它通过 `noteId` 关联笔记，永久删除笔记时级联删除。
- 字段：`noteId`、`anchorUtf16Offset`、`viewportOffsetFraction`、`updatedAt`、`deviceId`。
- 阅读滚动停止 450 ms 后保存；切换笔记或退出阅读界面时立即补记。
- 字符偏移使用 UTF-16 code unit，与 Swift `NSString` 和 Kotlin `String` 一致。
- 阅读位置不增加正文 `version`、不写正文历史、也不把笔记标记为 `dirty`。
- SSH 文件为 `<repositoryPath>/notes/reading_positions.json`，日期必须使用 ISO-8601；这点与普通笔记 JSON 使用的 Swift 2001 epoch 数字日期不同。
- 两端的新配置默认目录统一为 `~/notebook_backup`。Android 会只展开开头的 `~`，拒绝 `..`、控制字符和根目录，避免配置路径越界。
- Android 与 macOS 都先上传唯一临时文件并回读 SHA-256；附件和附件清单先发布，笔记 JSON 随后发布，`notes/index.json` 最后发布。删除仅通过墓碑隐藏，应用不自动物理清理远端旧 JSON/附件。
- Android 彻底删除前先持久化 `note|<id>` 墓碑；即使笔记实体已经从 Room 删除，下一次同步仍会发布删除状态，且墓碑不会因一次成功同步而清理，避免长期离线设备让笔记复活。
- 需要整体重建时，在 macOS 设置 → 数据管理执行“无损重建远程仓库”。成功切换到新目录后，让 Android 重新扫描 Mac 二维码或手工填入新路径；旧目录继续作为恢复快照保留。
- 冲突合并采用 LWW：`updatedAt` 较新者胜；时间相同则 `deviceID` 字典序较大者胜。
- 阅读位置写入远端前与远端现有值合并，并过滤已删除笔记。临时文件上传后做 SHA-256 回读校验，再原子改名。

### 阅读器交互与显示状态

- 每篇笔记保存 `viewMode`，支持 `text` / `preview` / `split` 协议值。Android 把 `split` 安全降级为预览；旧数据、导入文件和缺失字段默认 Markdown 预览，新建笔记默认编辑。
- 阅读器提供标题目录，手机以底部弹层显示；目录缩进保留标题层级，点击后跳到对应稳定块。
- 文档内搜索直接基于原始 Markdown 查找 UTF-16 位置，跳转到所在语义块，并在当前已渲染块中高亮可见文本。匹配结果设置上限，防止恶意或异常文档造成无界内存增长。
- 字号、行高和正文最大宽度是设备本地阅读偏好，使用 `sp` 并尊重系统字体缩放；设置持久化但不写入笔记正文和同步冲突模型。
- 正文居中并限制最大宽度；代码和表格仍保持各自的 Markdown 渲染语义。阅读列表右侧显示基于可见块的轻量滚动位置指示。
- Markdown 解析在 `Dispatchers.Default` 完成，切换笔记会自动取消旧解析；解析失败显示明确错误，不会无限停留在“正在加载笔记”。

协议示例：

```json
{
  "schemaVersion": 1,
  "exportedAt": "2026-07-16T12:00:00Z",
  "deviceID": "android-device-id",
  "positions": [
    {
      "noteID": "UUID",
      "anchorUTF16Offset": 1532,
      "viewportOffsetFraction": -0.02,
      "updatedAt": "2026-07-16T11:59:58Z",
      "deviceID": "android-device-id"
    }
  ]
}
```

### 保存与同步策略

- 每次正式保存后，仅当 SSH 的 host、username 和 path 已完整配置时，才排入一次受网络约束的即时同步。
- 即时任务使用唯一 WorkManager 工作并合并连续变更；失败按指数退避重试。15 分钟周期同步继续作为兜底。
- `dirty` 是持久化的待上传标记。同步会读取远端索引和库信息，但只为 `dirty` 笔记上传正文，不会每次重传全部笔记正文。
- 同步开始前先落盘所有待保存草稿；同步与保存均有互斥保护，远端确认只会清除同一版本的 `dirty`，不会误清除同步期间产生的新编辑。
- 正文、索引、库、附件清单和阅读位置均使用唯一临时文件；JSON 写入在改名前回读校验 SHA-256。
- SSH 主机密钥首次连接后固定；指纹变化必须由用户确认，不能静默接受。

### 提醒修复

- 笔记完成、删除或清除提醒时，同时取消 AlarmManager 任务和已经显示的通知。
- 每次排程都有随机 revision token。旧广播发现 token 已失效时直接退出，不能取消或覆盖更新后的提醒。
- 广播触发后从 Room 重新读取笔记，只有“未删除、未完成、仍有 reminderAt”才显示通知和排下一次每日/每周/月度提醒。
- 重复提醒重新排程时只替换闹钟，不会误删刚显示的通知；一次性提醒送达后只清理排程状态，保留通知供用户点击。

### 文件导入与加密目录移动

- 顶部导入按钮支持 `.txt`、`.md`、`.markdown`，导入到当前目录；当前目录为加密目录时同时建立加密映射。
- 单文件上限 50 MiB，后台分段读取；支持 UTF-8、UTF-16 LE/BE 和 Latin-1，统一换行为 `\n`，文件名作为标题。
- 加密笔记的目录菜单只列出加密目录，可以在加密目录之间移动；移动时更新目录元数据和加密映射并清空标签，不允许无意跨到普通目录。

## 关键回归用例

1. 创建/编辑笔记后，在已配置 SSH 时出现即时同步任务；未配置时不应报错或尝试联网。
2. 同步期间继续编辑同一笔记，新版本仍保持 `dirty`，不得被旧上传确认清除。
3. 在长笔记中滚动、切换到其他笔记再返回、强制结束并重启应用，位置均应恢复。
4. 两台设备分别产生阅读位置，新时间胜；相同时间按 `deviceID` 决胜，正文版本保持不变。
5. 完成一个每日提醒待办后，即使旧 Alarm 广播稍后到达，也不得显示通知或重新排程。
6. 导入 600 KiB Markdown 后列表滚动和笔记切换保持响应；列表查询不携带正文；代码围栏、表格、列表和标题目录均保持完整。
7. 修改阅读字号、行高和宽度后重启应用仍然保留；系统字体放大到 200% 时正文换行且页面本身不出现横向滚动。
8. 在文档内搜索靠近末尾的中文或 emoji，能跳到正确语义块；清除搜索后高亮完全移除。
9. 从一个加密目录移动笔记到另一个加密目录，同步后 macOS 与 Android 都只在目标目录显示。
10. 模拟远端临时文件回读哈希不一致，必须保留本地 `dirty` 并报错，不能覆盖正式远端文件。
